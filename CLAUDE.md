# Правила работы над Savio

Savio — десктопное GUI-приложение на Rust (`eframe`/`egui`), обёртка над внешними
`yt-dlp` и `ffmpeg`. Правила ниже обязательны при любом изменении кода.

## Архитектура: границы слоёв

Слои и направление зависимостей — часть контракта проекта, ломать нельзя:

| Слой | Файл | Что можно | Что нельзя |
|---|---|---|---|
| Домен | [src/model.rs](src/model.rs) | чистые типы, форматирование | знать про UI, про `yt-dlp`, про процессы |
| Движок | [src/engine/](src/engine/) | запуск процессов, разбор вывода | вызывать `egui`, знать про виджеты |
| UI | [src/app.rs](src/app.rs) | отрисовка, состояние экрана | парсить вывод `yt-dlp`, собирать аргументы CLI |

Связь движка с UI — только через канал `Event` и колбэк `notify`. Новый функционал
не должен добавлять других каналов связи: движок обязан оставаться пригодным для
CLI и тестов без единой правки.

---

## Правило 1. Всегда писать оптимизированный код

Не «быстрее любой ценой», а **без лишней работы и лишних аллокаций**.

**Конкретно для этого проекта:**

- **Не блокировать UI-поток.** `ui()` вызывается 60 раз в секунду. Любой ввод-вывод,
  запуск процесса или ожидание — только в отдельном потоке, результат приходит
  через `Event`. Никаких `.wait()`, `.output()`, чтения файлов внутри `ui()`.
- **Не аллоцировать в цикле отрисовки.** `format!`, `to_owned()`, `clone()`,
  `join()` внутри `ui()` выполняются каждый кадр. Если строка не меняется —
  считать её один раз и хранить в поле структуры, а не пересобирать в кадре.
- **Парсинг построчный, потоковый.** Вывод `yt-dlp` читается через `BufReader::lines()`
  по мере поступления. Не накапливать весь вывод в памяти, не читать в `String` целиком.
- **Логи ограничены.** `LOG_LIMIT` существует, чтобы длинная загрузка не съела память.
  Любой новый накопитель (история, очередь загрузок) обязан иметь такой же предел.
- **Заимствование вместо копирования.** Принимать `&str` / `&Path`, а не `String` /
  `PathBuf`, если владение не нужно. Возвращать срезы, где это возможно.
- **`clippy` — часть определения «оптимизированного».** Код считается готовым только
  после чистого `cargo clippy --all-targets -- -D warnings`.

Читаемость важнее микрооптимизаций. Ускорять то, что не в горячем пути (кадр
отрисовки, цикл парсинга), ценой ясности — не нужно.

## Правило 2. Новый функционал не ломает старый

Любое изменение обязано сохранить работоспособность того, что уже работает.

**Что считается «старым функционалом» и должно продолжать работать:**

- Скачивание MP4 в максимальном качестве и MP3 в максимальном качестве.
- Отмена загрузки: `Handle::cancel()` убивает процесс, UI показывает «Отменено»,
  а **не** ошибку.
- Прогресс-бар: с известным `total` — процент, с нулевым — неопределённый индикатор.
- Работа без `ffmpeg`: предупреждение в UI, приложение запускается и не падает.
- Отсутствие `yt-dlp`: понятное сообщение при старте, а не провал при первом клике.
- Поиск бинарников: сначала рядом с exe (портативная поставка), потом `PATH`.
- Кроссплатформенность: Windows, Linux, macOS. Каждая ветка `#[cfg]` имеет пару.
- Скрытие консоли на Windows (`hide_console`) при каждом запуске процесса.
- Успех без пути от `after_move` = файл уже существовал, это **не** ошибка.

**Правила совместимости:**

- Расширять `enum Event`, `Format`, `Line` — можно. Менять смысл существующих
  вариантов — нельзя. Все `match` по ним должны оставаться исчерпывающими без `_`,
  чтобы компилятор ловил пропущенные ветки.
- Аргументы `yt-dlp` (`--progress-template`, `--print after_move`) подобраны так,
  чтобы вывод оставался машинно-читаемым. Менять их — только вместе с `parse_line`.
- Комментарии в коде объясняют неочевидные решения (почему `|default` в шаблоне,
  почему события собираются перед применением в `drain_events`). Прежде чем менять
  такой участок — прочитать комментарий и понять причину.
- Появился баг-фикс или нетривиальная логика — добавить `#[test]`. Особенно на
  `parse_line`, `parse_media_info`, `human_bytes`, `human_duration`,
  `Progress::fraction`: это чистые функции, их легко покрыть.

**Проверка перед завершением задачи:**

```
cargo build
cargo clippy --all-targets -- -D warnings
cargo test
```

Все три должны пройти. Если тест падает — сообщить об этом прямо, а не замалчивать.

## Правило 3. Перед написанием кода — проверить зависимости

**Обязательная последовательность в начале работы над задачей:**

```
cargo outdated
```

Если обновления есть:

```
cargo update            # обновления в рамках semver, из Cargo.lock
cargo upgrade           # поднять сами требования версий в Cargo.toml
cargo build             # проверить, что собирается
cargo clippy --all-targets -- -D warnings
cargo test
```

**Дальше — обязательно:**

1. Посмотреть, **не сломалось ли что-то**. Не только компиляция — прогнать
   приложение и проверить сценарии из Правила 2.
2. Если сломалось — **адаптировать код под новую версию**, а не откатывать её.
   Откат допустим только если новая версия несовместима принципиально; в этом
   случае объяснить причину.
3. `eframe`/`egui` регулярно меняют API между минорными версиями (сигнатура
   `App::update`/`ui`, `ViewportBuilder`, конструкторы виджетов). Это самая
   вероятная точка поломки — проверять её первой.
4. Сообщить, что именно обновилось и что пришлось адаптировать.

Инструменты установлены: `cargo-outdated`, `cargo-edit` (даёт `cargo upgrade`).
Если команда не найдена — `cargo install cargo-outdated cargo-edit`.

## Общее

- Язык комментариев, сообщений в UI и текстов ошибок — русский, как в текущем коде.
- Сообщения об ошибках пишутся для пользователя, а не для разработчика: что
  случилось и что с этим делать.
- Edition проекта — 2024. Использовать современный синтаксис (`let ... else`,
  let-цепочки в `if`), как уже принято в коде.
- Не добавлять зависимость, если задача решается стандартной библиотекой.
  Текущий набор намеренно минимален: `eframe`, `rfd`, `serde_json`.
