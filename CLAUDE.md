# Правила работы над Savio

Savio — десктопное GUI-приложение на Rust (`eframe`/`egui`), обёртка над внешними
`yt-dlp` и `ffmpeg`. Правила ниже обязательны при любом изменении кода.

## Архитектура: границы слоёв

Слои и направление зависимостей — часть контракта проекта, ломать нельзя:

| Слой | Файл | Что можно | Что нельзя |
|---|---|---|---|
| Домен | [src/model.rs](src/model.rs) | чистые типы, форматирование | знать про UI, про `yt-dlp`, про процессы |
| Движок | [src/engine/](src/engine/) | запуск процессов, разбор вывода, сеть | вызывать `egui`, знать про виджеты |
| UI | [src/app.rs](src/app.rs) | отрисовка, состояние экрана | парсить вывод `yt-dlp`, собирать аргументы CLI |
| Оформление | [src/theme.rs](src/theme.rs) | палитра, `Style`, метрики | знать про состояние загрузки, про `Event`, про движок |

Связь движка с UI — только через канал `Event` и колбэк `notify`. Новый функционал
не должен добавлять других каналов связи: движок обязан оставаться пригодным для
CLI и тестов без единой правки.

## Правило 1. Всегда писать оптимизированный код

Не «быстрее любой ценой», а **без лишней работы и лишних аллокаций**.

**Конкретно для этого проекта:**

- **Не блокировать UI-поток.** `ui()` вызывается 60 раз в секунду. Любой ввод-вывод,
  запуск процесса или ожидание — только в отдельном потоке, результат приходит
  через `Event`. Никаких `.wait()`, `.output()`, чтения файлов внутри `ui()`.
- **Не аллоцировать в цикле отрисовки.** `format!`, `to_owned()`, `clone()`,
  `join()` внутри `ui()` выполняются каждый кадр. Если строка не меняется —
  считать её один раз и хранить в поле структуры, а не пересобирать в кадре.
- **Парсинг построчный, потоковый.** Вывод `yt-dlp` читается через `BufReader::lines()`
  по мере поступления. Не накапливать весь вывод в памяти, не читать в `String` целиком.
- **Логи ограничены.** `LOG_LIMIT` существует, чтобы длинная загрузка не съела память.
  Любой новый накопитель (история, очередь загрузок) обязан иметь такой же предел.
- **Заимствование вместо копирования.** Принимать `&str` / `&Path`, а не `String` /
  `PathBuf`, если владение не нужно. Возвращать срезы, где это возможно.
- **`clippy` — часть определения «оптимизированного».** Код считается готовым только
  после чистого `cargo clippy --all-targets -- -D warnings`.

Читаемость важнее микрооптимизаций. Ускорять то, что не в горячем пути (кадр
отрисовки, цикл парсинга), ценой ясности — не нужно.

## Правило 2. Новый функционал не ломает старый

Любое изменение обязано сохранить работоспособность того, что уже работает.

**Что считается «старым функционалом» и должно продолжать работать:**

- Скачивание MP4 в максимальном качестве и MP3 в максимальном качестве.
- Отмена загрузки: `Handle::cancel()` убивает процесс, UI показывает «Отменено»,
  а **не** ошибку.
- Прогресс-бар: с известным `total` — процент, с нулевым — неопределённый индикатор.
- Работа без `ffmpeg`: предупреждение в UI, приложение запускается и не падает.
- Отсутствие `yt-dlp`: понятное сообщение при старте, а не провал при первом клике.
- Поиск бинарников: сначала рядом с exe (портативная поставка), потом `PATH`,
  потом каталог данных. Каталог данных **последний**: системный `ffmpeg`
  пользователя должен выигрывать у скачанного нами.
- Установка при первом запуске: всё найдено — модалки нет и запуск мгновенный;
  сбой загрузки `ffmpeg` — не ошибка, а предупреждение (без него Savio работает);
  сбой загрузки `yt-dlp` — ошибка, но приложение всё равно открывается.
  Модалку всегда можно закрыть кнопкой: запирать пользователя в окне нельзя.
- Кроссплатформенность: Windows, Linux, macOS. Каждая ветка `#[cfg]` имеет пару.
  Ветки не под текущую ОС не компилируются локально — их проверяет CI на трёх
  системах, поэтому матрицу в [ci.yml](.github/workflows/ci.yml) не сокращать.
- Скрытие консоли на Windows (`hide_console`) при каждом запуске процесса.
- Успех без пути от `after_move` = файл уже существовал, это **не** ошибка.

**Правила совместимости:**

- Расширять `enum Event`, `Format`, `Line` — можно. Менять смысл существующих
  вариантов — нельзя. Все `match` по ним должны оставаться исчерпывающими без `_`,
  чтобы компилятор ловил пропущенные ветки.
- Аргументы `yt-dlp` (`--progress-template`, `--print after_move`) подобраны так,
  чтобы вывод оставался машинно-читаемым. Менять их — только вместе с `parse_line`.
  **Префикс в шаблоне (`download:`, `postprocess:`, `after_move:`) выбирает момент
  вывода и самим `yt-dlp` съедается** — в поток приходит голый JSON без него.
  Поэтому `parse_line` различает шаблоны по полям (`event`, `pp`, `downloaded`),
  а не по началу строки. Новый шаблон обязан иметь своё опознаваемое поле.
  Ровно на этом прогресс-бар молча не работал: строки прогресса уходили в журнал.
- Комментарии в коде объясняют неочевидные решения (почему `|default` в шаблоне,
  почему события собираются перед применением в `drain_events`). Прежде чем менять
  такой участок — прочитать комментарий и понять причину.
- Появился баг-фикс или нетривиальная логика — добавить `#[test]`. Особенно на
  `parse_line`, `parse_media_info`, `human_bytes`, `human_duration`,
  `Progress::fraction`: это чистые функции, их легко покрыть.

**Проверка перед завершением задачи:**

```
cargo build
cargo clippy --all-targets -- -D warnings
cargo test
```

Все три должны пройти. Если тест падает — сообщить об этом прямо, а не замалчивать.

**Для изменений в UI этого мало.** Сборка не ловит визуальные баги: выключенную
кнопку, неотличимую от включённой; полосу шапки, не дотянутую до края; нечитаемый
текст. Прогнать приложение и посмотреть глазами состояния: покой, отсутствие
`yt-dlp`, невалидная ссылка, загрузка, готово, отмена. Отдельно — минимальный
размер окна (520×420): содержимое не должно обрезаться.

И последнее — по Правилу 5: не устарел ли `README.md` и не появилось ли по ходу
работы правило, которое стоит записать сюда.

## Правило 3. Перед написанием кода — проверить зависимости

**Обязательная последовательность в начале работы над задачей:**

```
cargo outdated
```

Если обновления есть:

```
cargo update            # обновления в рамках semver, из Cargo.lock
cargo upgrade           # поднять сами требования версий в Cargo.toml
cargo build             # проверить, что собирается
cargo clippy --all-targets -- -D warnings
cargo test
```

**Дальше — обязательно:**

1. Посмотреть, **не сломалось ли что-то**. Не только компиляция — прогнать
   приложение и проверить сценарии из Правила 2.
2. Если сломалось — **адаптировать код под новую версию**, а не откатывать её.
   Откат допустим только если новая версия несовместима принципиально; в этом
   случае объяснить причину.
3. `eframe`/`egui` регулярно меняют API между минорными версиями (сигнатура
   `App::update`/`ui`, `ViewportBuilder`, конструкторы виджетов). Это самая
   вероятная точка поломки — проверять её первой. Проверять **по исходникам
   в реестре** (`~/.cargo/registry/src/*/egui-*/src/`), а не по памяти: имена
   и типы меняются молча. В 0.35 это `CornerRadius::same(u8)` и `Margin::same(i8)`
   (не `f32`), `Panel` вместо исчезнувшего `TopBottomPanel`, `set_style_of(Theme, …)`
   вместо `set_style`. Каждое из этих отличий — ошибка компиляции на ровном месте.
   Хуже те, что компилируются: `ViewportBuilder::with_maximized(true)` вместе с
   `with_inner_size` выставляет окну **признак** развёрнутого, не меняя саму
   геометрию, — `IsZoomed` отвечает «развёрнуто», а окно остаётся прежнего
   размера. И из-за этого признака `ViewportCommand::Maximized(true)` на первом
   кадре становится пустой операцией: разворачивать, по мнению системы, нечего.
   Разворачивать окно надо **или** флагом, **или** командой, но не обоими сразу.
4. Сообщить, что именно обновилось и что пришлось адаптировать.

Инструменты установлены: `cargo-outdated`, `cargo-edit` (даёт `cargo upgrade`).
Если команда не найдена — `cargo install cargo-outdated cargo-edit`.

## Правило 4. Оформление — только через `theme.rs`

Тема — тёмный фон с жёлтым акцентом. Палитра проверена по контрасту, и любая
правка обязана эту проверку сохранить.

- **Ни одного `Color32::from_rgb` вне `theme.rs`.** В `app.rs` — только именованные
  константы темы. Новый цвет — сначала константа с ролью, потом использование.
- **Новая пара «текст на фоне» — посчитать контраст** по WCAG 2.1 и записать
  коэффициент в комментарий рядом с константой. Порог: 4.5:1 для текста,
  3:1 для границ и крупного текста. Не проходит — менять цвет, а не порог.
- **На жёлтом акценте текст только тёмный.** Жёлтый светлый: белым по нему —
  около 1.4:1, нечитаемо. Тёмная подпись даёт 12.4:1.
- **Тема тёмная принудительно** (`ThemePreference::Dark`). По умолчанию egui
  следует за системной схемой, и на светлой ОС стиль просто не применился бы,
  а палитра на светлом фоне по контрасту не проверялась.
- **Состояния (hover / active / disabled) задавать через `visuals`,** а не
  фиксированной заливкой: `Button::fill` по документации egui отключает реакцию
  на наведение — кнопка выглядит мёртвой. Выключенный вид задавать явно:
  `ui.disable()` не переключает виджет на `noninteractive`, а только глушит
  прозрачность, поэтому выключенная кнопка легко становится неотличимой от
  включённой.
- **Стиль собирается один раз** при старте (`theme::apply`), а не в кадре:
  `Style` содержит `BTreeMap` шрифтов, пересобирать его 60 раз в секунду незачем.

## Правило 5. Версия и документация обновляются вместе с кодом

Код, версия и документация расходятся молча: ошибки сборки это не даёт,
замечают через месяцы. Поэтому в конце задачи — три вопроса.

**1. Поднята ли версия?**

Версия поднимается **в конце каждой задачи**, менять её отдельным заходом
«перед релизом» не нужно. Причина простая: пользователь сообщает о проблеме
и называет версию из шапки окна. Если под одним номером ходят три разные
сборки, это сообщение бесполезно — воспроизводить нечего.

Править **только `version` в [Cargo.toml](Cargo.toml)**, это единственный
источник правды. Шапка приложения берёт версию через `env!("CARGO_PKG_VERSION")`,
`Info.plist` заполняет CI из того же места. Второй копии числа в проекте быть
не должно — копии расходятся, и замечают это уже после выпуска.

Насколько поднимать (semver; пока версия 0.x, мажор не трогаем):

- **patch** (`0.2.0` → `0.2.1`) — исправление, рефакторинг, правка текстов;
- **minor** (`0.2.0` → `0.3.0`) — новая возможность или заметное изменение
  поведения: то, о чём есть что написать пользователю.

Выпуск делается тегом `v<версия>`, совпадающим с `Cargo.toml`.

**2. Не устарел ли `README.md`?**

Обновлять, если изменение заметно пользователю:

- появилась, исчезла или заметно изменилась возможность;
- изменились требования: внешние программы, поддерживаемые ОС, установка;
- изменился порядок работы из раздела «Использование».

Не трогать при рефакторинге и внутренних правках: README описывает продукт,
а не устройство кода.

**3. Не пора ли добавить правило сюда?**

Добавлять, если по ходу задачи выяснилось то, на чём споткнётся следующий:

- молчаливое поведение библиотеки или внешней программы — то, что не ловится
  ни компилятором, ни тестами (как съеденный префикс `--progress-template`);
- решение, неочевидное из кода, которое легко «починить» обратно и тем сломать;
- новый слой или файл со своими ограничениями — тогда и строку в таблицу слоёв.

**Не добавлять:** то, что и так видно из кода; подробности одной конкретной
задачи; то, что уже описано — сначала перечитать файл и поискать дубль. Правила
ценны, пока их читают целиком; разросшийся файл перестают открывать.

Формулировать коротко и **с причиной**: правило без «почему» через полгода
выглядит произволом, и его нарушают первым.

О новой версии и о каждой правке в `README.md` и в этом файле — сказать в отчёте
по задаче отдельной строкой. Правило, добавленное молча, легко пропустить и трудно
оспорить: последнее слово о том, становится ли частный случай общим правилом, —
за человеком. С версией так же: только человек знает, тянет изменение на minor
или это patch.

## Правило 6. Внешняя программа, вернувшая 0, могла ничего не сделать

Правило 2 уже ловит один такой случай (съеденный префикс `--progress-template`).
Ниже — те же грабли у инструментов, которые появились вместе с установкой
зависимостей. Общее у них одно: **ошибки не будет ни при сборке, ни в тестах,
ни в коде возврата** — просто перестанет работать функция.

- **`tar` завершается с кодом 0, даже если не распаковал ни одного файла.**
  Достаточно ошибиться в глубине `--strip-components`: шаблоны не совпадут,
  tar честно отчитается об успехе, каталог останется пустым. Проверено вживую.
  Поэтому после распаковки **всегда** `Path::exists()` на каждый ожидаемый
  файл; код возврата — не доказательство.
- **`tar` из `PATH` на Windows — почти наверняка не тот `tar`.** Git for
  Windows, MSYS2 и Cygwin кладут в `PATH` свой GNU tar раньше системного, а
  GNU tar не читает zip вообще. Системный `bsdtar` звать по абсолютному пути
  `%SystemRoot%\System32\tar.exe`. На macOS и Linux `tar` из `PATH` — верный.
- **Ключ `--wildcards` асимметричен**: GNU tar (Linux) без него понимает имена
  членов архива буквально и по шаблону не находит ничего, а bsdtar (Windows,
  macOS) на него падает с «Option --wildcards is not supported». Две ветки
  `#[cfg]`, и обе обязательны.
- **Формат архива выбирается под распаковщик, а не по вкусу.** Штатный bsdtar
  в Windows 10 собран без liblzma и `.tar.xz` не понимает, хотя `tar --help`
  показывает ключ `-J` — по справке поддержку не определить. Отсюда zip для
  Windows и macOS, `.tar.xz` только для Linux.
- **`ureq` с фичей `gzip` (она включена по умолчанию) возвращает `None` из
  `Body::content_length()`.** Прогресс-бар молча выродится в неопределённый
  индикатор. Читать заголовок вручную бесполезно: там размер сжатых данных,
  и полоса перескочит 100%. Отсюда `default-features = false`.

## Общее

- Язык комментариев, сообщений в UI и текстов ошибок — русский, как в текущем коде.
- Сообщения об ошибках пишутся для пользователя, а не для разработчика: что
  случилось и что с этим делать.
- Edition проекта — 2024. Использовать современный синтаксис (`let ... else`,
  let-цепочки в `if`), как уже принято в коде.
- Не добавлять зависимость, если задача решается стандартной библиотекой.
  Текущий набор намеренно минимален: `eframe`, `rfd`, `serde_json`.
